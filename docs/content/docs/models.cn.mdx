## 文本（代理）

`/v1/models` 列出以下 chat 模型。cli2api 根据 model 名形态**自动路由**到两套
不同的上游 surface，客户端不用配置。

| 厂商 | 模型 | 路由到 |
|------|------|--------|
| OpenAI（传统 chat） | `gpt-5`、`gpt-5-mini`、`gpt-5-nano`、`gpt-4.1`、`gpt-4.1-mini`、`o4-mini` | `/v1/chat/completions` |
| OpenAI（code-plane） | `openai/gpt-5.5`、`openai/gpt-5.4-mini`、`openai/gpt-5.4-nano`、`openai/gpt-5.3-codex`† | `/vendors/openai/v1/chat/completions` |
| DeepSeek | `deepseek-v4-flash`、`deepseek-v4-pro` | `/v1/chat/completions` |
| Anthropic | `claude-sonnet-4-6`、`claude-haiku-4-5`、`claude-opus-4-7` | `/v1/chat/completions`（`/v1/messages` 用 x-api-key） |

带 `vendor/` 前缀的 model 会切到 code-plane（`mulerun code` / opencode 用来跑
GPT-5.x 新模型族的那条路径）。前缀在转发前会从 body 里**剥掉**，因为上游路径里
已经编码了 vendor。

† `openai/gpt-5.3-codex` 是 **Responses API** 模型——调用走 `/v1/responses`，
不是 `/v1/chat/completions`。cli2api 自动把 Responses 调用代理到
`/vendors/openai/v1/responses`。

```sh
# OpenAI SDK 指向 cli2api —— 通过 code-plane 跑 gpt-5.5：
export OPENAI_BASE_URL=http://localhost:51222/v1
export OPENAI_API_KEY=<你的 cli2api inbound key>
codex --model openai/gpt-5.5
```

<Callout type="info" title="GPT-5.x thinking 模式吃 token">
GPT-5.x 模型族在出可见内容前会把 `max_tokens` 大部分用在 reasoning 上。如果
`message.content` 回空且 `finish_reason: length`，把 `max_tokens` 调到 500+，
看 `usage.completion_tokens_details.reasoning_tokens` 知道思考用了多少。
</Callout>

<Callout type="warn" title="Vendor 路由的边界情况">
- **Anthropic 模型**（`anthropic/claude-*` 等）在 `/v1/messages` 上**还没**接
  code-plane 路由——对应上游 URL 没验过。先用不带前缀的 `claude-opus-4-7` 这种。
- **压缩请求**（Content-Encoding: gzip/br/deflate）会跳过 vendor 路由，因为
  cli2api 无法解析 body 看 model 字段。带 vendor 前缀的请求要关掉客户端压缩。
- **Body > 64MB** 返回 HTTP 413（`request_too_large`），跟全局 RequestSize
  中间件同一个上限。
</Callout>

<Callout type="warn" title="不做别名映射">
cli2api **不**把 OpenAI 模型名映射到 MuleRun。请用 MuleRun 真名
（`gpt-image-2`、`wan2.6-t2i`、`midjourney`）——不是 `dall-e-3`。别名会让客户端
误以为在调 OpenAI，行为差异很难排查。
</Callout>
